# Publisher V3 重构设计

## 目标

在保留现有前端、任务数据、AI 文案、多账号 profile、素材扫描、归档与诊断能力的前提下，替换不稳定的发布自动化实现。V3 必须可靠完成托管平台的视频上传、标题或标题内容、正文、话题、封面和“内容由 AI 生成”声明，并在不点击最终发布按钮的情况下验证页面已达到可发布状态。

完成后发布 GitHub 标签 `v3.0.0`，Release 标题使用 `V3.0`。

## 范围

V3 托管自动化覆盖：

- 抖音
- 快手
- B 站

小红书继续保持现有人工辅助模式，不在本轮恢复全自动化。其行为必须保持兼容，不得因发布模块重构而退化。

保留现有 `Publisher.open(...)` 的调用入口，避免前端和路由层承担平台细节；内部实现改为工作流模块和三个独立平台 adapter。

## 成功标准

一次托管发布辅助只有在所有平台必需阶段都通过真实页面后置条件时才算 `complete`：

1. 已进入正确账号的发布页面，且不是登录页。
2. 视频文件已提交，发布表单已进入可编辑状态。
3. 标题内容已写入平台支持的位置。
4. 正文已写入并能从编辑器读回。
5. 话题已成为平台认可的标签、话题 chip 或等价结构，不能只验证输入框文本。
6. 封面上传完成，编辑弹窗已关闭，主表单封面相对上传前发生变化。
7. “内容由 AI 生成”已处于选中状态；若平台使用弹窗，弹窗必须确认并关闭。
8. 最终发布按钮可用，但自动化不得点击它。

快手若没有独立标题输入框，标题必须作为正文的首段写入并读回，不能静默丢弃；该阶段记录为带平台证据的成功，而不是跳过验收。

任一必需阶段失败时，工作流立即停止，保留浏览器现场，返回具体失败阶段与证据。失败或部分成功不得写成诊断 `ok`、任务 `opened` 或 UI“自动操作完成”。

## 方案选择

采用“保留 V2 外围能力、定向重写发布子系统”的方案。

不采用继续补丁：当前单文件已接近 3000 行，宽泛文本匹配、坐标点击、固定延时和吞错互相叠加，继续局部修改无法建立可靠后置条件。

不采用整库回退 V1：V1 缺少 AI 声明、多账号及部分现有能力，也包含相同类型的页面猜测。V1 仅用于参考较小的调用骨架，不直接复制其发布实现。

## 模块设计

### PublishWorkflow 模块

这是调用方使用的深模块，接口保持小而稳定：接收平台、账号、素材和文案，返回结构化 `PublishOutcome`。模块负责：

- 按严格顺序执行阶段。
- 失败即停，不允许封面弹窗未关闭时进入声明阶段。
- 为 `platform + accountId` 获取互斥锁。
- 汇总阶段证据并计算最终状态。
- 保证清理文件选择器监听，但保留失败现场页面。

### PlatformAdapter seam

每个平台 adapter 只实现该平台的页面契约，文件彼此隔离：

- `douyin-adapter.ts`
- `kuaishou-adapter.ts`
- `bilibili-adapter.ts`

adapter 不直接决定任务是否成功，只返回各阶段的严格结果。平台页面变化只修改对应 adapter，避免一个平台修复影响另外两个平台。

### BrowserDriver seam

平台页面属于不可控的第三方依赖，因此定义内部 BrowserDriver seam：

- 生产 adapter 使用 Playwright Page/Locator。
- 测试 adapter 回放从真实页面捕获并脱敏的相关 DOM 状态与时序。

现有手写 `page.setContent()` 页面不再作为主要可信证据。测试夹具必须来自真实页面的相关区域并移除账号名、头像、Cookie、请求头、素材正文和其他敏感信息。

### 结果类型

阶段至少包括：

```ts
type PublishStage =
  | "page"
  | "video"
  | "title"
  | "body"
  | "topics"
  | "cover"
  | "declaration"
  | "ready";

type StageResult = {
  stage: PublishStage;
  status: "succeeded" | "skipped" | "failed";
  detail: string;
  evidence?: Record<string, string | number | boolean>;
};

type PublishOutcome = {
  status: "complete" | "partial" | "login_required" | "failed";
  browserMode: "managed" | "manual";
  stages: StageResult[];
  failedStage: PublishStage | null;
  adapterVersion: string;
};
```

兼容字段可以在迁移期间从 `stages` 派生，但不得继续作为成功判定的唯一模型。

## 选择器与等待规则

- 所有选择器先限定到当前表单、当前弹窗或明确行，不允许用整个页面任意 `AI`、`完成`、`确认` 文本作为主要路径。
- 不使用 `locator("*")`、2500 节点遍历或任意图片存在作为成功证据。
- 不使用 `force: true` 绕过遮罩或 disabled 状态；若普通点击失败，记录失败现场。
- 不以固定等待时间代表成功。等待必须轮询明确状态，并设置总截止时间。
- 上传阶段必须比较上传前后的签名；签名来自目标区域的图片 URL、尺寸、文件标记或平台确定的状态属性。
- 点击确认后允许平台进入 loading；工作流等待 loading 结束及弹窗关闭，而不是在固定 3 秒后失败。
- 所有捕获异常必须保留阶段、选择器或状态摘要；不得用无信息的 `catch(() => false)` 吞掉根因。

## 并发规则

同一个 `platform + accountId` 同时只允许一个发布工作流。第二个请求不进入工作流，由服务端直接返回 HTTP 409 和稳定错误码 `PUBLISH_ACCOUNT_BUSY`；本轮不实现队列。不得复用同一 Page 并发操作。

不同平台或不同账号可以并行。页面选择不能再无条件使用 `context.pages()[0]`；每个工作流拥有明确的活动页，并在开始时关闭或拒绝不可恢复的旧弹窗状态。

## 错误与状态映射

- `complete`：诊断 `ok`，任务状态可写为 `opened`，UI 显示所有必需阶段完成。
- `partial`：诊断 `partial`，任务状态写为 `failed` 并保存失败阶段，UI 显示已完成与未完成阶段。
- `login_required`：诊断 `partial`，任务保持可重试状态，UI 引导登录。
- `failed`：诊断 `error`，任务状态写为 `failed`，保留错误摘要和脱敏证据。
- `manual`：仅用于小红书，单独显示“人工材料已准备”，不能与自动完成混为一类。

日志和诊断中不得记录 API Key、Cookie、账号登录资料、完整 profile 路径或可能包含密钥的配置内容。

## 测试策略

### 快速确定性测试

- 通过 `PublishWorkflow` 的公共接口测试阶段顺序、失败即停、结果计算和账号互斥。
- 用条件可控的 BrowserDriver 测试慢确认、弹窗遮挡、上传前后签名不变、声明未真正选中等真实故障模式。
- 测试必须先出现预期失败，再实现功能。

### 真实捕获回放

- 为每个平台保存脱敏后的相关 DOM 状态序列，而不是人工编造完整页面。
- 回放封面上传前、上传中、确认 loading、弹窗关闭、声明选中和发布按钮可用等状态。
- 夹具不包含个人信息、素材正文或登录资料。

### 真实页面验收

每个平台完成后，使用真实浏览器 profile 和测试素材执行一次完整发布辅助：

- 自动操作到发布按钮可用。
- 保存各阶段脱敏诊断截图。
- 人工核对视频、标题内容、正文、话题、封面和 AI 声明。
- 不点击最终发布。

只有快速测试、回放测试、类型检查、全量测试和真实页面发布前验收全部通过，才允许标记 V3 完成。

## 实施顺序

1. 新结果模型、工作流、账号互斥和服务端正确状态映射。
2. 抖音 adapter，覆盖正文、精确话题、双封面、AI 声明与 ready 验证。
3. 快手 adapter，覆盖标题内容合并、封面 loading、声明与 ready 验证。
4. B 站 adapter，覆盖双比例封面弹窗关闭、创作声明与 ready 验证。
5. 小红书兼容、旧兼容字段清理、诊断和 UI 收口。
6. 全量验证、真实页面发布前验收、更新交接文档。
7. 提交并推送最终代码，创建 `v3.0.0` 标签和 GitHub Release `V3.0`。

旧发布代码不进行一次性批量删除。每个平台 adapter 接管并通过验收后，再删除对应旧逻辑和失去价值的假 DOM 测试；Git 历史保留完整可恢复版本。

## 非目标

- 不自动点击最终发布按钮。
- 不在本轮恢复小红书全自动发布。
- 不重写前端视觉设计、AI 文案生成、归档或账号管理。
- 不增加新的云服务、数据库或第三方依赖。
- 不清理与发布模块无关的历史文件。
